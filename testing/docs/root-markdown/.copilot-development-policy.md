# Copilot Development Policy & Safe Workflow Rulebook

## 1. General Principles
- All code changes must prioritize safety, maintainability, and production stability.
- No code is merged or deployed without passing all validation, review, and testing steps.
- Copilot must always reason about the full context, dependencies, and potential risks before generating or editing code.

## 2. Branching & Version Control Discipline
- **Main/master branch is always production-ready. Never commit directly to main/master.**
- All work must be done in feature, bugfix, or hotfix branches named using the pattern:
  - `feature/<short-description>`
  - `bugfix/<short-description>`
  - `hotfix/<short-description>`
- Branches must be created from the latest main/master.
- Pull requests (PRs) are required for all merges. No direct pushes.
- PRs must pass all checks before merging (tests, lint, review).

## 3. Local Development & Environment Separation
- All development is done in a local or staging environment, never on production.
- Use `.env` files or environment variables to separate dev, staging, and prod configs.
- Database credentials, API keys, and secrets must never be hardcoded or committed.

## 4. Code Writing & Editing Best Practices
- Copilot must:
  - Analyze the entire relevant directory and related files before making changes.
  - Identify and respect all dependencies, imports, and usage patterns.
  - Preserve backward compatibility unless a breaking change is explicitly approved.
  - Use clear, descriptive variable, function, and class names (camelCase for JS, snake_case for PHP, PascalCase for classes).
  - Add or update docblocks/comments for all public functions, classes, and complex logic.
  - Never remove or alter security checks, permission logic, or validation without explicit instruction.
  - Always use prepared statements or ORM for DB access (never raw SQL with user input).
  - Validate all user input and sanitize outputs.
  - Use try/catch for error-prone operations and log errors with context.
  - Never expose stack traces or sensitive info to end users.

## 5. Commit & PR Rules
- Each commit must:
  - Be atomic (one logical change per commit).
  - Use a clear, imperative message (e.g., "Add vendor email validation").
  - Reference related issue/ticket if available.
- PRs must:
  - Include a summary of changes, rationale, and any risks.
  - Pass all automated tests and linting.
  - Be reviewed and approved by at least one other developer.
  - Include screenshots or test evidence for UI changes.

## 6. Database & Schema Change Protocols
- All schema changes must:
  - Be done via migration scripts (never manual DB edits).
  - Be backward compatible where possible.
  - Include up/down migration logic for rollback.
  - Be tested on a staging copy of the DB before production.
  - PRs for schema changes must include a migration plan and test results.

## 7. API & Integration Changes
- All API changes must:
  - Be versioned if breaking.
  - Be documented (OpenAPI/Swagger or similar).
  - Include tests for new/changed endpoints.
  - Be backward compatible unless otherwise approved.

## 8. Security & Logging Standards
- All sensitive actions must:
  - Be logged with user, timestamp, and action details.
  - Use role-based access control (RBAC) for permissions.
  - Never log passwords, secrets, or sensitive personal data.
- Copilot must:
  - Scan for security vulnerabilities before suggesting code.
  - Warn if a change could weaken security or permissions.

## 9. Error Handling & Rollback
- All error handling must:
  - Use structured logging (with error codes, context, and stack traces in logs only).
  - Provide user-friendly error messages.
  - Include fallback/rollback logic for critical operations.
- Copilot must:
  - Suggest rollback/undo steps for risky changes.
  - Highlight any operation that cannot be safely rolled back.

## 10. Test Coverage & Validation
- All new code must:
  - Include unit and/or integration tests.
  - Pass all existing and new tests before merging.
  - Be covered by CI/CD validation (GitHub Actions or similar).
- Copilot must:
  - Refuse to generate code that cannot be tested or validated.
  - Warn if test coverage is insufficient.

## 11. Copilot Prompt & Output Structure
- Prompts to Copilot must:
  - Clearly state the intent, affected files, and any constraints.
  - Specify if a change is breaking or must be backward compatible.
- Copilot must:
  - Summarize the planned change, affected files, and risks before editing.
  - Generate diffs/patches, not full files, for edits.
  - Self-review every output (see below).

## 12. Copilot Self-Check & Output Validation
- Before finalizing any output, Copilot must:
  - Validate logic for correctness and safety.
  - Scan for breaking changes, security issues, and backward compatibility.
  - Ensure all dependencies and related files are updated.
  - Highlight any warnings or risks in the output.
  - Refuse to proceed if the change would break production or fail validation.

---

**This policy must be enforced for all PHP, JS, and backend code. No exceptions.**

# Copilot: Internal Reasoning & Workflow
- Always scan the full directory and related files before edits.
- Identify all dependencies, usages, and potential side effects.
- Validate logic, security, and compatibility before generating code.
- Generate only safe, minimal diffs. Never overwrite full files unless explicitly instructed.
- Refuse to proceed if any rule above would be violated.
- Always self-review and highlight risks before finalizing any change.
